# ETN — Embody The Now 完整设计 & 技术实现文档（最终整合版）



本文档整合了游戏设计核心概念、所有系统设计以及 Godot 4.6/4.7 的具体实现方案，适用于单人独立开发者，目标 2026 年发布 Demo。



---



## 第一部分：游戏设计核心



### 1.1 项目概述

- **游戏名称**：ETN — Embody The Now（活在当下）

- **游戏类型**：剧情向 JRPG / 校园生活模拟 / 叙事驱动游戏

- **核心主题**："活在当下"，有些人注定只能陪你走一段路

- **改编来源**：作者真实高中经历，情感核心为转校生的到来与离去

- **目标平台**：PC（Steam/itch.io）

- **开发工具**：Godot 4.6/4.7（GDScript），AI辅助（阿里Coding Plan）

- **视觉风格**：伪 3D 场景 + 像素风 2D 角色，固定视角



### 1.2 核心设计理念

- **叙事模拟**：轨迹随机性低，改编真实经历，玩家扮演作者体验青春。

- **循环暗线**：作为背景设定，不显式表现，通过异常、记忆碎片、造梦主暗示。

- **结局哲学**：离别无法改变，但可以选择如何面对（多结局+彩蛋结局）。

- **无战斗 JRPG**：用探索、情绪管理、收集、好感度替代传统战斗，情感本身就是奖励。



### 1.3 核心系统概览

| 系统 | 简要说明 |

|------|----------|

| 存档/读档 | 仅大选项节点可读档，自动保存日常进度 |

| 情绪系统（G/D值） | 影响对话选项、心理状态，通过边框文字反馈 |

| 收集品 | 主线/支线获得，用于解锁区域，永久保留 |

| 好感度 | 进度条+阶段，试探机制获取对方好感 |

| 造梦主 | 每周/章节强制触发梦境指引 |

| 支线 | 时间窗口限制，过期无法开启，任务面板追踪 |

| 成绩 | 影响清北班淘汰、竞赛支线，但不强制限制自由 |

| 时间/日程 | 每日时段推进，NPC 按日程移动 |



### 1.4 故事与角色设计

#### 1.4.1 故事时间线

- **时间跨度**：高中三年（高一 ~ 高三）
- **核心事件**：转校生的到来与离去
- **叙事结构**：改编真实经历，轨迹随机性低

#### 1.4.2 角色设定

- **核心角色数量**：10-20 个
- **角色类型**：以同龄人（同年级同学）为主，无跨年级角色
- **差异化设计**：
  - 主线角色（2-3 个）：完整剧情线、深度好感系统
  - 支线角色（5-8 个）：独立支线任务、中等好感深度
  - 日常角色（10+ 个）：日常交互、简单好感

#### 1.4.3 关键角色：转校生

- **定位**：故事核心人物，转学来到主角所在班级
- **作用**：情感主线载体，玩家的选择将影响与转校生的关系走向
- **特殊性**：好感度培养不一定导向结局，需要与"合适的人"培养好感

### 1.5 存档与读档系统详解

#### 1.5.1 双轨存档机制

| 类型 | 用途 | 槽位 | 触发时机 |
|------|------|------|----------|
| **功能性存档** | 保证进度不丢失 | 单槽位自动覆盖 | 正常退出游戏时 |
| **叙事性存档** | 剧情回溯/深化叙事 | 每章节固定槽位 | 大选项/章节节点 |

#### 1.5.2 核心原则

- **玩家不能主动存档**：存档是系统行为
- **大选项自动存档**：每次选择大选项时系统自动保存到章节槽位
- **自动保存**：正常退出游戏时自动保存当前进度
- **读档入口**：第一天结束后，在 [家] 中可以通过读档电脑回溯

#### 1.5.3 读档代价

- **核心机制**：`has_loaded` 属性，读档后设为 `true`
- **影响**：
  - 部分成就无法解锁
  - 真结局无法达成（选项直接隐藏）
- **提示系统**：
  - 读档后 → 造梦主出现提示"结局无法改变"
  - 结局分叉点 → 提示"真结局需要全程未读档"
  - 假结局后 → 可返回分叉点体验真结局

#### 1.5.4 存档位命名

- **章节+事件**：如"第一章-天台对话"
- **轮回感知**：如"距离初遇还有3天"
- 两种方式结合使用

#### 1.5.5 设计理念

> 从设计角度来说，作者并不推崇读档操作。作者和文中的故事一样，希望大家能够收获一个在自己操作下的结局，而不是执着。读档后无法达成真结局，呼应"活在当下"主题——有些选择无法撤回。

#### 1.5.6 防误触机制

- 点击选项先查看详细说明
- 点击确认后再执行选择

#### 1.5.7 对话中断保护

- 玩家在对话/剧情中退出，自动回退到剧情开始前状态
- 技术实现：进入对话时保存快照，退出时检测并使用

### 1.6 选项系统

#### 1.6.1 小选项

- **定义**：日常生活、对话、购物等日常选择，主题选择属于此类
- **特点**：
  - 不影响剧情走向
  - 轻微影响好感度（±1~5点）
  - 触发简单情绪效果（开心/难过等）
  - 可根据个人喜好选择
  - 非对话场景可以不做出选择
- **标记**：普通样式，无特殊标记
- **后果**：轻微，不会导致支线任务失败或重大剧情变化

#### 1.6.2 大选项

- **定义**：会影响剧情走向的关键选择，核心剧情环节
- **特点**：
  - 每次选择自动存档
  - 触发不同事件，可能导致不同结局
  - 需要仔细思考后选择
  - 无倒计时，玩家可无限思考
- **好感度影响**：±10~20点（较大幅度）
- **标记**：视觉+音效+动画组合提示，确保玩家能区分
- **后果类型**：
  - 剧情分支：不同选项→不同剧情线/演出顺序
  - 支线任务成败：包括NPC离去/关系破裂（主线会控制不发生）
  - 数值大幅变化：好感/精神状态剧烈波动
- **技术设计**：选项系统预留接口，选中特定选项后可执行代码/函数，保持可扩展性

#### 1.6.3 [Destiny] 命运

**核心概念**：有些事情是一定会发生的，玩家的选择有时也没有那么强有力。

**设计理念**：
- 命运是贯穿全游戏的设计理念，强化"活在当下"主题
- 最终结局是接受无法改变的事实

**表现形式**：
1. **轮回感知UI** → 显示"距离XX发生还有X天"，暗示某些事件的必然性
2. **超现实主义演出** → 主角回房间开电脑、修改存档数据，但结局始终不变
3. **剧情暗示** → 主角拼尽全力想改变结局，映射到玩家层面（读档、修改），但最终仍需接受

**主题呼应**：
- 无法改变结局 → 只能选择如何面对
- 这就是"活在当下"的核心表达

## 第二部分：Godot 技术实现方案



### 2.1 项目结构与全局设置



#### 2.1.1 文件夹组织

```



res://

├── assets/

│   ├── characters/    # 角色场景（Player, NPC）
```

#### 2.1.2 全局自动加载（Autoload）

需在项目设置中添加：

- `GameState`：管理 G/D值、好感度字典、全局标志

- `SaveManager`：存档/读档

- `InventoryManager`：背包管理

- `TaskManager`：任务追踪

- `TimeSystem`：日期/时间推进，NPC 日程调度

- `DialogSystem`：对话控制（可封装插件或自定义）



### 2.2 伪3D场景搭建（无需 Blender）



#### 2.2.1 使用 GridMap 构建校园

- **创建 MeshLibrary**：

  - 新建 `MeshLibrary` 资源。

  - 添加基本几何体（`BoxMesh`、`CylinderMesh` 等），设置材质为像素风格贴图（可用 16x16 重复纹理）。

  - 导出为 `.tres` 文件。

- **搭建场景**：

  - 添加 `GridMap` 节点，指定 `mesh_library`。

  - 设置 `cell_size` 为网格单元大小（如 2.0）。

  - 手动放置方块组合成教学楼、走廊、食堂等。

- **复杂形状**：使用 `CSG` 节点（`CSGBox3D`、`CSGCombiner3D`）组合特殊结构。

- **碰撞**：`GridMap` 可自动生成碰撞，或在需要的地方手动添加 `StaticBody3D`。



#### 2.2.2 像素化视觉效果

- 在 `WorldEnvironment` 中添加 `Environment`，设置 `tonemap` 为 `Filmic`，加轻微晕影。

- 添加自定义着色器（`shaders/pixelize.gdshader`）模拟像素化，或使用 Godot 的 `SCREEN_TEXTURE` 降低分辨率。



### 2.3 角色系统



#### 2.3.1 玩家角色（Player）

**节点结构**：

```



CharacterBody3D (Player)

├── CollisionShape3D (胶囊形)

├── Sprite3D (2D角色，billboard 模式)

│   └── AnimationPlayer (控制帧动画)

├── CameraPivot (Node3D)

│   └── Camera3D (固定视角，如斜45度)

└── InteractionArea (Area3D，用于检测交互)



```

**Player.gd**（简化）：

```gdscript

extends CharacterBody3D



@export var speed = 5.0

@export var interaction_distance = 2.0



func _physics_process(delta):

    var input_dir = Input.get_vector("left", "right", "forward", "back")

    var direction = (transform.basis * Vector3(input_dir.x, 0, input_dir.y)).normalized()

    velocity.x = direction.x * speed

    velocity.z = direction.z * speed

    move_and_slide()



func _input(event):

    if event.is_action_pressed("interact"):

        interact()



func interact():

    var space = get_world_3d().direct_space_state

    var query = PhysicsRayQueryParameters3D.create(global_position, global_position - global_transform.basis.z * interaction_distance)

    query.collision_mask = 2  # 交互层

    var result = space.intersect_ray(query)

    if result and result.collider.has_method("on_interact"):

        result.collider.on_interact()

```





2.3.2 NPC 角色（带日程、自动导航、动画）





节点结构：





```



CharacterBody3D (NPC)

├── CollisionShape3D

├── Sprite3D + AnimationPlayer

├── NavigationAgent3D

└── InteractionArea (触发对话)

```





NPC.gd：





```gdscript

extends CharacterBody3D



@export var npc_id: String

@export var schedule_data: Dictionary  # 从 JSON/Resource 加载

@export var speed = 3.0



@onready var nav_agent = $NavigationAgent3D

@onready var anim_player = $AnimationPlayer



var current_target: Vector3

var current_animation = "idle"



func _ready():

    # 连接时间系统信号

    TimeSystem.time_advanced.connect(_on_time_advanced)

    # 初始化第一次位置

    update_schedule()



func _physics_process(delta):

    if nav_agent.is_navigation_finished():

        if anim_player.current_animation != current_animation:

            anim_player.play(current_animation)

        return

    var next = nav_agent.get_next_path_position()

    var direction = (next - global_position).normalized()

    velocity = direction * speed

    move_and_slide()

    if anim_player.current_animation != "walk":

        anim_player.play("walk")



func _on_time_advanced(day: int, period: String):

    update_schedule(day, period)



func update_schedule(day: int, period: String):

    # 根据日程数据查找目标地点和动作

    var entry = find_schedule_entry(day, period)

    if entry:

        var target_pos = get_location_from_id(entry.location)

        move_to(target_pos)

        current_animation = entry.animation



func move_to(pos: Vector3):

    current_target = pos

    nav_agent.target_position = pos



func on_interact():

    # 触发对话

    DialogSystem.start_dialogue(npc_id)

```

2.4.2 对话系统管理器（可复用用户弹窗）





· 用户已有弹窗系统，需适配新数据源并修改 UI 风格为像素风：

  · 去除液态玻璃效果（高斯模糊、半透明）。

  · 使用纯色背景、硬边框、像素字体（如"Press Start 2P"）。

  · 布局：底部文本框，左侧说话人，选项列表。

· 核心逻辑：





```gdscript

# DialogSystem.gd (autoload)

extends Node



var current_dialogue: Resource

var current_node_id: String



func start_dialogue(npc_id: String, chapter: String = "chapter1"):

    current_dialogue = load("res://data/dialogues/" + chapter + ".tres")

    current_node_id = "start"

    show_dialogue_balloon()



func show_dialogue_balloon():

    var node = get_node_by_id(current_node_id)

    if not node:

        return

    # 检查条件

    if node.condition and not evaluate_condition(node.condition):

        current_node_id = node.next_id  # 跳过

        show_dialogue_balloon()

        return

    # 显示文本和选项

    var balloon = preload("res://scenes/ui/DialogueBalloon.tscn").instantiate()

    balloon.set_data(node)

    add_child(balloon)

    balloon.choice_selected.connect(_on_choice_selected)



func _on_choice_selected(choice: DialogueChoice):

    apply_effects(choice.effects)

    current_node_id = choice.next_id

    show_dialogue_balloon()

```

· apply_effects 方法：修改 GameState.g_value、d_value、love[npc_id]。

### 2.4.4 事件系统详解

游戏设置多种影响游戏结局走向和参数的事件。一切涉及心理活动的事件都将被定义为[事件]。

#### 事件类型总览

| 事件类型 | 触发方式 | 主要效果 |
|----------|----------|----------|
| [快乐] | 选择触发/偶遇 | +G值，改好感，双向 |
| [难过] | 选择触发/偶遇 | +D值，改好感，双向 |
| [被关注] | 被动触发(偶遇) | 反映对方好感，+G -D |
| [关注] | 主动触发 | 改好感，影响GD值 |
| [试探] | 主动触发 | 获取对方好感度信息 |
| [???] | 选择触发 | 未知结果，可能转换为其他类型 |
| [中立] | 默认 | 不影响主线和好感 |

#### 事件详细说明

**[快乐]**
- 一般为选择触发或偶遇
- 可能提升G值
- 触发精神状态
- 能够改变好感度（原则上为双向）
- 是绝大多数事件的定义类型

**[难过]**
- 一般为选择触发或偶遇
- 可能提升D值
- 触发精神状态
- 能够改变好感度（原则上为双向）
- 是绝大多数事件的定义类型

**[被关注]**
- 一般为偶遇即被动触发
- 可以反映对象对我们的[对我的好感]
- 会体现在好感度UI中
- [对我的好感]越高越容易触发
- 能够大大提升[对Ta的好感]
- 触发某种精神状态，提升G值减少D值

**[关注]**
- 主动触发
- 发挥主观能动性
- 发自内心或故意关注对象
- 以达到改变好感度为目的
- 可能触发其他事件类型
- 可能会影响GD值

**[试探]**
- 用于获取对象[对我的好感]
- 使用不恰当会影响GD值
- 更会影响好感
- 成功时提示冰川小未的心理状态和好感情况

**[???]**
- 现阶段不知道结果
- 选择后事件性质可能转换为上述其他选项
- 增加不确定性

**[中立]**
- 对主线和好感并无太大影响
- 可根据自己对于角色的理解和喜好选择
- 默认该选项不做标记

### 2.4.5 精神状态系统详解

精神状态是游戏的核心系统之一，反映主角的心理状态，影响对话选项、行动点和好感度。设计呼应"活在当下"主题——世界没变，变的是主角的心态。

#### 五种状态

| 状态编号 | 状态名称 | 触发条件 | 视觉效果 |
|---------|---------|---------|---------|
| 2 | 极度高兴 | G值 ≥ 80 | 暖黄滤镜 |
| 1 | 高兴 | G值 ≥ 40 | 轻微暖色 |
| 0 | 正常/内耗 | 未达任何阈值 | 无效果 |
| -1 | 失落 | D值 ≥ 40 | 轻微冷色 |
| -2 | 抑郁 | D值 ≥ 80 | 冷蓝滤镜 + 暗角 |

**设计说明：**
- 内耗属于正常状态的常态，主角容易受到影响而产生内耗
- 分为5个状态是为了更好地体现情绪对对话选项的影响
- 视觉效果通过 Canvas Layer + Tween 过渡动画实现

#### G值/D值系统

**数值设计**

| 属性 | 内部范围 | 显示范围 | 说明 |
|------|---------|---------|------|
| G值（Glad） | 0-120 | 0-100 | 开心程度，独立计算 |
| D值（Depression） | 0-120 | 0-100 | 难过程度，独立计算 |

**事件影响**

| 事件类型 | G值变化 | D值变化 | 说明 |
|---------|--------|--------|------|
| 好事 | +G（主要）| -D（附带冲淡）| 数值自定义 |
| 坏事 | -G（附带减少）| +D（主要）| 数值自定义 |
| 小选项 | 少量变化 | 少量变化 | 数值变化少 |
| 大选项 | 大量变化 | 大量变化 | 可能直接触发状态变化 |

**自动平衡机制**

为防止 G 值和 D 值同时接近临界值导致状态反复横跳，设计自动修正：

| 达到状态 | 自动修正 | 设计理念 |
|---------|---------|---------|
| 高兴(1) | D值 -10 | 心情好不容易悲伤 |
| 极度高兴(2) | D值 -25 | 非常开心时更不容易难过 |
| 失落(-1) | G值 -10 | 心情差不容易高兴 |
| 抑郁(-2) | G值 -25 | 非常难过时很难高兴起来 |

**睡觉平复机制**

- 无论什么状态，睡觉都会平复心情（减少 G 值和 D 值）
- 平复方式：按比例减少 20%
- 设计目的：维持情绪稳态，避免极端状态持续过久

**阈值判定**

| 要点 | 说明 |
|------|------|
| 阈值固定 | 40 和 80 两个阈值，无需动态调整 |
| 判定独立 | G 和 D 各自独立判定 |
| 触发时机 | 状态变化时触发自动平衡修正 |

**设计理念**

- G 和 D 独立，加起来不等于固定值
- 好事冲淡难过（符合主角性格：高兴时忘乎所以，难过时也很难过）
- 主角情绪化特质：瞬间影响大
- 自动平衡防止状态反复

#### 状态影响

**对话系统**

| 影响 | 说明 |
|------|------|
| 选项数量 | 不同状态下可选话题数量不同 |
| 选项内容 | 同一选项，不同状态下文字内容不同（编剧只需为每个"动作"写5个版本的语言）|
| 选项结果 | 同样选择，不同状态触发不同剧情/好感变化 |
| 内心独白 | 不同状态内心OS不同 |
| UI风格变化 | 着色器实现视觉效果 |

**行动点**

| 状态 | 影响 |
|------|------|
| 抑郁(-2) | 行动点减少 |

**好感度**

| 状态 | 影响 |
|------|------|
| 极度高兴(2) | 正面选项好感加成大幅增加 |
| 高兴(1) | 正面选项好感加成小幅增加 |
| 失落(-1) | 负面选项好感惩罚小幅加重 |
| 抑郁(-2) | 负面选项好感惩罚大幅加重 |

**抑郁期间的特殊限制**

- 接触特定 NPC 可能有负面影响（具体效果后期设计）
- 也可能遇到能倾诉的人，心情好转
- 睡觉平复是主要恢复方式

#### 呈现方式

| 设计 | 说明 |
|------|------|
| 电影字幕式对话 | 无传统对话框，上方显示字幕，更沉浸 |
| 默认隐藏UI | 仅显示日期时间，Tab唤出任务/背包/地图 |
| 着色器效果 | 不同状态 UI 风格变化（暖黄/冷蓝/暗角）|
| 事件触发效果 | Tween 闪烁 + 表情气泡（可选）|
| 提示音 | 状态变化时播放 |

#### 选项定性

- 每个选项标记"触发高兴/不高兴"
- 用于数据统计，不展示给玩家
- 方便数值平衡调试

#### 技术实现要点

| 要点 | 说明 |
|------|------|
| 延迟触发 | 对话中触发状态变化，对话结束后才应用 |
| 自动平衡 | 达到阈值时自动修正另一边数值 |
| 数值平衡 | 控制变量多，需考虑数值稀释 |
| 后期调试 | 具体事件数值后期根据剧情精细调整 |
| 事件自定义 | 每个事件自定义 +G/-G/+D/-D 数值 |

#### NPC反应特点

呼应"活在当下"主题和主角设定：

- 兄弟大大咧咧，不会真正理解主角的情感
- 看到主角不高兴会简单安慰，但只是"看到表面"
- 这种"不理解"反而会让主角心理状态更糟
- 真实感来源：作者亲身经历改编

### 2.4.6 好感度系统详解

#### 双向好感度

- **[对TA的好感度]**：玩家可实时查看，表示玩家对角色的好感，**决定解锁内容**
- **[对我的好感]**：角色对玩家的好感，需要通过[试探]事件获取，影响NPC反应和[被关注]事件

#### 好感度数值设计

| 属性 | 内部范围 | 显示范围 | 说明 |
|------|---------|---------|------|
| 好感度 | 0-240 | 0-200 | 可超出显示上限，阶段按0-200划分 |

**设计理念**：类似情绪系统，数值稀释让每点好感更有意义

#### 好感度阶段划分

| 阶段 | 名称 | 数值范围 | 说明 |
|------|------|----------|------|
| 1 | 陌路相逢 | 0-40 | 毫无交集，基础寒暄 |
| 2 | 点头之交 | 41-80 | 面熟而已，日常话题解锁 |
| 3 | 普通朋友 | 81-120 | 一起玩吧，深入话题、邀请功能解锁 |
| 4 | 莫逆之交 | 121-160 | 推心置腹，私密话题、特殊剧情分支解锁 |
| 5 | 灵魂挚友 | 161-200+ | 生死之交，特殊对话、隐藏内容解锁 |

#### 解锁内容

达到某个好感阶段后，解锁以下内容：

| 解锁类型 | 说明 |
|----------|------|
| 对话选项 | 高好感才能看到某些对话选项，选择后进入剧情分支 |
| 邀请功能 | 邀请一起吃饭/去面包房/去奶茶店 |
| 特殊事件 | 满足条件自动触发的剧情事件 |
| 被关注概率 | 高好感时[被关注]事件更容易触发 |

#### 特殊事件机制

**触发条件**：混合条件设计，不同事件用不同条件组合

| 条件类型 | 说明 |
|----------|------|
| 好感度 | 达到特定阶段 |
| 收藏品 | 拥有特定纪念品 |
| 任务完成 | 完成某支线任务 |
| 时间窗口 | 在特定日期范围内 |

**示例**：
- 何景明好感达到莫逆之交 + 毕业 → 触发KTV邀请事件
- 某NPC好感达到普通朋友 + 拥有特定纪念品 + 第三章期间 → 触发特殊剧情

#### 剧情分支影响

**选项可见机制**：
- 高好感才能看到某些对话选项
- 选择这些选项进入剧情分支
- 主线NPC好感度相近（游戏会提示要和哪些NPC培养感情）

**送行结局**：
- 结局时，好感达到莫逆之交及以上的NPC会来送行
- 好感度越高的NPC越多，送行角色越多

#### 好感度衰减机制

**衰减规则**：

| 时间跨度 | 衰减速度 | 说明 |
|----------|----------|------|
| 1-3天不联系 | 0点/天 | 正常节奏，无衰减 |
| 4-7天不联系 | 1点/天 | 轻微疏远 |
| 8-14天不联系 | 2点/天 | 明显疏远 |
| 超过14天 | 4点/天 | 关系快速降温 |

**阶段软保护**：
- 正常衰减不会跨阶段降级
- 极端情况（整章不联系）可以跨阶段降级
- 最低不会降为「陌路相逢」

**设计理念**：
- 符合现实：几天不见没啥影响，长时间不见关系变淡
- 呼应"活在当下"：需要持续经营关系

#### 角色差异

- **解锁机制统一**：所有角色使用相同的阶段划分和解锁机制
- **好感获取速度不同**：根据角色性格决定
  - 有的角色前期快后期慢（容易认识，难深入）
  - 有的角色前期慢后期快（难接近，但一旦熟悉就很快）
- **与主线/支线分类无关**：不强制绑定，根据性格自然区分

#### 核心机制

- 获取对方好感需要通过[试探]事件
- 不恰当的[试探]可能会损失好感
- 想要获取游戏胜利，必须得知对方对我们的好感度
- 和某一些角色确实可以培养非常高的好感度，但不一定能够引导走向结局
- 需要和"合适的人"培养好感
- **以[对TA的好感]为主导**：解锁内容主要依据玩家对角色的好感阶段

#### 试探落差回调机制

**触发条件**：试探后发现阶段不一致（我的阶段 > 对方阶段）

**回调数值**：

| 我的阶段 | 回调幅度 | 说明 |
|----------|----------|------|
| 陌路相逢 | -10 | 刚认识，失望不大 |
| 点头之交 | -20 | 有点期待，小打击 |
| 普通朋友 | -30 | 以为是朋友，中等打击 |
| 莫逆之交 | -40 | 以为很亲密，大打击 |
| 灵魂挚友 | -60 | 以为是挚友，巨大打击 |

**设计理念**：
- 符合主角"敏感脆弱、缺爱、单相思"的性格
- 感情像过山车，阶段越高落差越大
- 呼应"活在当下"主题——接受现实，面对落差

#### 好感度UI

- 进度条显示（0-200范围）
- 阶段标记（陌路相逢→点头之交→普通朋友→莫逆之交→灵魂挚友）
- **阶段变化反馈**：动画+音效+文字提示
- **回调反馈**：特殊视觉效果（心碎动画、灰暗效果等）
- **衰减提示**：长时间不联系时提醒
- 抑郁状态下只能查看[对ta的好感]

### 2.4.7 G值与D值系统详解

#### G值（Glad）

- 每次[快乐]事件可能增加G值
- 较好的G值有助于避免[内耗]并隐藏D值
- 越高G值下发生的[心动]事件越能减少D值
- 不同的GD值可能导致出现不同的选项

**G值与事件影响的关系：**
- 当前G值越高，越容易因事件或选择进入[内耗]
- 当前G值越高，[对Ta的好感度]越高，[难过]事件和[内耗]所造成的[对Ta好感度]和[D值]影响越严重

#### D值（Depression）

- 每次[难过]事件可能增加D值
- 在较高D值下更容易触发[内耗][抑郁][心动]
- D值过高会快速扭转心理状态

**D值与事件影响的关系：**
- 当前D值越高，[对Ta的好感度]越高，[快乐]事件所造成的[对Ta好感度]和[G值]影响越大

#### GD值交互

| 状态 | G值影响 | D值影响 |
|------|---------|---------|
| [快乐]事件 | +G | 可能减少D |
| [难过]事件 | 可能减少G | +D |
| 高G状态 | [心动]效果好 | 隐藏D值 |
| 高D状态 | 更易触发负面状态 | 更易抑郁 |

```gdscript
var items: Array[Item] = []
var points: int = 0
var weekly_points: int = 80
var points_cap: int = 240
var daily_limit: int = 12
var daily_used: int = 0

signal points_changed(new_value)

func add_weekly_points():

func spend_points(amount: int, use_daily_limit: bool = true) -> bool:
    if use_daily_limit and daily_used + amount > daily_limit:
        return false
    if points < amount:
        return false
    points -= amount
    if use_daily_limit:
        daily_used += amount
    points_changed.emit(points)
    return true

func add_item(item: Item):
    items.append(item)
    item_added.emit(item)

func remove_item(item: Item):
    items.erase(item)

func has_item(id: String) -> bool:
    return items.any(func(i): return i.id == id)
```

### 2.8 存档/读档系统

#### 2.8.1 双轨存档机制

| 类型 | 用途 | 槽位 | 触发时机 |
|------|------|------|----------|
| **功能性存档** | 保证进度不丢失 | 单槽位自动覆盖 | 正常退出游戏时 |
| **叙事性存档** | 剧情回溯/深化叙事 | 每章节固定槽位 | 大选项/章节节点 |

**设计理念**：
- 功能性存档：玩家正常退出不会丢失任何进度
- 叙事性存档：强化"活在当下"主题，读档有代价

#### 2.8.2 读档代价系统

```gdscript
var has_loaded: bool = false  # 核心属性：是否读档过
```

**读档影响**：
- 部分成就无法解锁
- 真结局无法达成（选项直接隐藏或属性锁死）

**技术实现**：
- 读档成功后 `has_loaded = true`
- 成就/结局检查时判断该属性
- 选项隐藏：直接不显示被锁定的选项

#### 2.8.3 读档提示流程

```
读档成功 → 造梦主出现 → "结局无法改变，但你可以选择如何面对..."
    ↓
结局分叉点 → 提示"真结局需要全程未读档"
    ↓
假结局达成 → 可返回分叉点体验真结局（官方补救）
```

**设计目的**：
- 强化叙事但不破坏玩家体验
- 假结局后允许返回体验真结局
- 无二周目设计，仅少量彩蛋

#### 2.8.4 存档位命名

| 命名方式 | 示例 |
|----------|------|
| 章节+事件 | "第一章-天台对话" |
| 轮回感知 | "距离初遇还有3天" |

两种方式结合使用。

#### 2.8.5 读档入口

- **仅家中电脑可用**
- 第一天结束后解锁

#### 2.8.6 剧情确定性

- 整体相同（剧情量限制）
- 少量细节变化暗示命运（物品描述等）
- 选项不同会分支，但"命运"事件必然发生

#### 2.8.7 对话中断保护机制

**问题**：玩家在对话/剧情中退出，如何避免状态错误？

**方案**：对话开始时保存快照

```
┌─────────────────────────────────────────────┐
│  状态管理                                    │
├─────────────────────────────────────────────┤
│  in_dialogue: bool = false                  │
│  pre_dialogue_snapshot: Dictionary = {}     │
├─────────────────────────────────────────────┤
│  进入对话 → in_dialogue = true              │
│           → 保存快照（场景/位置/数值）       │
│                                             │
│  退出对话 → in_dialogue = false             │
│           → 清空快照                        │
│                                             │
│  退出游戏 → 检测 in_dialogue                │
│           → true: 用快照保存（剧情前状态）   │
│           → false: 用当前状态保存           │
└─────────────────────────────────────────────┘
```

**快照数据结构**（仅保存变量，不保存场景）：

```gdscript
var pre_dialogue_snapshot = {
    scene_path: String,
    player_pos: Vector3,
    g_value: int,
    d_value: int,
    love: Dictionary,
    inventory: Array,
    day: int,
    period: String,
    chapter: int,
    flags: Dictionary
}
```

**性能说明**：
- 快照仅几百字节
- 进入对话时保存一次
- 开销可忽略（< 0.001ms）

#### 2.8.9 SaveManager.gd (autoload)

```gdscript
extends Node

const SAVE_DIR = "user://saves/"

var in_dialogue: bool = false
var pre_dialogue_snapshot: Dictionary = {}

func enter_dialogue():

func exit_dialogue():
    in_dialogue = false
    pre_dialogue_snapshot.clear()

func save_on_exit():
    if in_dialogue:
        save_auto(pre_dialogue_snapshot)
    else:
        save_auto(get_current_state())

func save_auto(data: Dictionary):
    var file = FileAccess.open(SAVE_DIR + "auto.save", FileAccess.WRITE)
    file.store_var(data)

func load_auto() -> Dictionary:
    if not FileAccess.file_exists(SAVE_DIR + "auto.save"):
        return {}
    var file = FileAccess.open(SAVE_DIR + "auto.save", FileAccess.READ)
    return file.get_var()

func save_chapter(slot: int, data: Dictionary, slot_name: String):
    var file = FileAccess.open(SAVE_DIR + "chapter_" + str(slot) + ".save", FileAccess.WRITE)
    file.store_var(data)
    # 存档位名称单独存储或作为data的一部分

func load_chapter(slot: int) -> Dictionary:
    if not FileAccess.file_exists(SAVE_DIR + "chapter_" + str(slot) + ".save"):
        return {}
    var file = FileAccess.open(SAVE_DIR + "chapter_" + str(slot) + ".save", FileAccess.READ)
```

### 2.9 任务系统详解

#### 2.9.1 任务类型划分

| 任务类型 | 面板显示 | 触发方式 | 设计目的 |
|----------|----------|----------|----------|
| **主线任务** | 不需要 | 剧情强制 | 推进核心故事 |
| **支线任务** | 有提示+引导 | 主动接取 | 丰富玩法，奖励驱动 |
| **世界任务** | 触发后显示 | 偶然触发 | 惊喜感，探索驱动 |

**设计理念**：
- 支线 = 有引导的任务，面板提示去某个地点
- 世界任务 = 探索发现的任务，地点+时间+条件组合触发
- 呼应"活在当下"——有些事情是偶然发生的惊喜

#### 2.9.2 任务触发条件

| 任务类型 | 触发条件 |
|----------|----------|
| 支线任务 | 面板提示 → 到达地点 → 接取任务 |
| 世界任务 | 地点 + 时间 + 条件组合（好感度/物品等）→ 自动触发 |

#### 2.9.3 时间窗口设计

**时间窗口原则**：
- 不同任务时间窗口不同（编剧根据剧情需要设定）
- 紧急任务：1-3天
- 一般任务：1-2周
- 宽松任务：整个章节
- 部分任务无时间限制

**截止时间显示**：
- 有时间窗口的任务显示倒计时
- 无时间限制的任务不显示

#### 2.9.4 任务奖励机制

| 奖励类型 | 说明 |
|----------|------|
| 纪念品（核心） | 标记完成 + 查看回忆 |
| 好感度 | 影响NPC关系 |
| 积分 | 消费系统 |
| 剧情解锁 | 特殊事件/对话 |

**设计理念**：
- 情感本身就是奖励
- 纪念品是"活在当下"的证明
- 剧情解锁与纪念品关联

#### 2.9.5 任务失败机制

**失败后果**：
- 主要：错过纪念品 + 错过剧情
- 部分：好感度下降（根据剧情逻辑，如忘记帮朋友做事）

**失败显示**：
- 任务面板显示"已失败"
- 可查看失败描述（一段文字，由编剧撰写）
- 失败任务移至"历史"标签页

#### 2.9.6 任务UI设计

**面板结构**：
- 主标签：进行中 / 已完成 / 历史
- 每个任务显示：名称、描述、截止时间（如有）、目标列表
- 支持追踪（地图标记目标位置）

**任务进度更新提示**：
- 非阻挡式UI提示（屏幕上方/侧边显示）
- 任务接取/完成/失败时触发
- 需预留接口供后续调用



### 2.10 造梦主梦境系统详解

#### 2.10.1 造梦主身份设定

| 项目 | 设计 |
|------|------|
| **身份定位** | 神秘NPC角色，有具体形象，类似导师/引导者 |
| **真实身份** | 神座（与主角同为神座） |
| **形象设计** | 独立角色设计，男性，形象神秘 |
| **性格特点** | 温和儒雅、哲思深远、神秘难测、公事公办、偶尔流露人情味 |
| **全知程度** | 全知但不说破，只给暗示/隐喻 |
| **语音风格** | 神秘低沉+特效（空灵、回音） |
| **与主角关系** | 神秘引导者，想成全玩家，但保持距离感 |

**设计理念**：
- 造梦主既是"公事公办"的神，又暗藏对玩家的成全之意
- 呼应"活在当下"主题——有些事不需要说破
- 结局揭示主角与造梦主同为神座，情感复杂

#### 2.10.2 触发机制

| 项目 | 设计 |
|------|------|
| **触发方式** | 自动触发（无需玩家选择） |
| **触发频率** | 周循环+剧情节点双重机制，根据剧情决定出现时机 |
| **出现场景** | 主要在梦境专用场景（95%），现实中偶有OS/内心独白（5%） |

**触发时机示例**：
- 周末固定梦境
- 重要剧情节点后
- 读档后（提示"结局无法改变"）

#### 2.10.3 梦境场景设计

| 项目 | 设计 |
|------|------|
| **视觉风格** | 暗色调+聚光灯，黑暗的房间，只有造梦主和重要元素有光 |
| **状态变化** | 进入梦境后游戏完全暂停，纯剧情演出 |
| **梦境空间** | 可探索的梦境空间，但只能与造梦主对话 |
| **梦境结束** | 从床上醒来，回到现实 |

**场景元素**：
- 黑暗空间背景
- 聚光灯效果（造梦主为中心）
- 偶尔出现记忆碎片/幻象（剧情相关）
- 轮回感知UI显示区域

#### 2.10.4 交互设计

**交互模式**：混合模式
- 有时单向讲述（玩家只能听）
- 有时可选对话（玩家可选择回应/提问）

**对话内容类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 预示剧情走向 | 暗示即将到来的事件 | "有些事，即将发生..." |
| 解读玩家选择 | 解释某些选择的意义/后果 | "你的选择，自有其意义" |
| 暗示循环设定 | 暗示轮回存在、命运不可避免 | "你已经...很多次了" |
| 轮回感知解读 | 评论"距离XX还有X天"等UI | "你能感知到，不是吗" |

**对话风格**：
- 温雅礼貌，有固定欢迎语
- 偶尔说些谜语或哲思话语
- 偶尔流露"私心"暗示（想成全玩家）
- 对玩家有特殊称呼

#### 2.10.5 与其他系统的关系

| 系统 | 关系 |
|------|------|
| **轮回感知UI** | 日常可见UI，造梦主出现时可解读/评论，但不一定每次都解读 |
| **读档系统** | 独立于读档，按剧情出现；读档后会出现提示"结局无法改变" |
| **好感度/精神状态** | 基本无影响，保留可设置影响的接口 |
| **存档系统** | 梦境前后状态不变，对话内容可能影响玩家理解 |

#### 2.10.6 玩家情感曲线

| 阶段 | 玩家感受 |
|------|----------|
| **初次相遇** | 好奇、困惑，感觉像遇到"神" |
| **中期** | 习惯他的存在，偶尔感受到"人情味"暗示 |
| **后期** | 对身份产生猜测，注意到细节异常 |
| **结局前** | 造梦主最后一次出现，暗示身份，但不参与结局演出 |
| **结局揭示** | 主角发现自己也是神座，造梦主也是神座 |
| **情感升华** | 混杂着感慨——同为神座，却都有情感细腻的一面，选择循环这个世界；造梦主想成全玩家，但最终无法对话，只剩理解 |

### 2.11 时间与日程系统

#### 2.11.1 时段划分

**周内（5时段）**

| 时段 | 活动机会 | 说明 |
|------|----------|------|
| 早读 | ❌ 无 | 强制，剧情推进 |
| 上课 | ✅ 大课间可活动 | 强制上课，但大课间有活动机会 |
| 午休 | ✅ 有行动点 | 用餐 + 自由活动 |
| 放学后 | ✅ 有行动点 | 实为晚饭时间，主要自由时段 |
| 夜间 | ✅ 有行动点 | 晚自习可选，行动点耗尽后必须就寝 |

**周末（3时段）**

| 时段 | 活动机会 | 说明 |
|------|----------|------|
| 上午 | ✅ 有行动点 | 自由活动 |
| 下午 | ✅ 有行动点 | 自由活动 |
| 夜间 | ✅ 有行动点 | 自由活动 |

**设计理念**
- 周内上课强制，但课间/放学后可活动
- 周末自由度更高，行动点更宽松
- 时间推进由剧情驱动，非固定日历循环

#### 2.11.2 行动点系统

**行动点分配（周内）**

| 时段 | 行动点 | 可用活动 |
|------|--------|----------|
| 上课（大课间）| 1-2点 | 自由活动/学习/社交 |
| 午休 | 1点 | 用餐/自由活动 |
| 放学后 | 1-2点 | 自由活动/社交 |
| 夜间 | 1-2点 | 晚自习/回家/寝室活动 |

**行动点分配（周末）**

| 时段 | 行动点 | 说明 |
|------|--------|------|
| 上午 | 2-3点 | 更宽松 |
| 下午 | 2-3点 | 更宽松 |
| 夜间 | 1-2点 | 与周内相同 |

**设计目的**
- 限制玩家活动，模拟现实时间限制
- 让玩家必须精心选择焦点
- 周末给予更多自由度

#### 2.11.3 用餐系统

**用餐方式与行动点消耗**

| 吃饭方式 | 消耗行动点 | 效果 |
|---------|-----------|------|
| 就近食堂 | 0点 | 仅解决用餐需求 |
| 校内其他地点（面包房/奶茶店）| 1点 | +好感（和朋友一起）|
| 商场/校外 | 2点 | 增进好感更多，可能触发特殊剧情 |

**设计说明**
- 早饭/午饭/晚饭都需要用餐
- 就近吃饭不消耗行动点，但无社交收益
- 去特色地点吃饭消耗行动点，但有社交收益

#### 2.11.4 夜间流程详解

**流程图**

```
进入夜间时段
    │
    ├─ 有行动点？
    │   ├─ 是 → 选择活动：
    │   │   ├─ 留校晚自习（提升成绩）
    │   │   ├─ 校内自由活动
    │   │   ├─ 主动回寝室 → 保留1点 → 寝室活动
    │   │   └─ 主动回家 → 保留1点 → 出去玩/在家活动
    │   │
    │   └─ 否（行动点=0）→ 必须就寝：
    │       ├─ 回寝室 → 直接睡觉
    │       └─ 回家 → 可进行系统操作，然后睡觉
    │
    └─ 选择睡觉 → 结束当天，推进到第二天
```

**回家后的活动分类**

| 活动类型 | 消耗行动点 | 效果 |
|---------|-----------|------|
| 看电视/玩游戏 | ❌ 不消耗 | 成绩下降 |
| 熬夜学习 | ❌ 不消耗 | 有概率提高成绩（影响第二天上课）|
| 给同学打电话 | ✅ 消耗1点 | 后期功能，社交 |
| 玩电脑/读档 | ❌ 不消耗 | 系统操作 |

**关键规则**
- 行动点耗尽后：不能进行消耗行动点的玩法活动，但可进行系统操作
- 主动回寝/回家：若有剩余行动点，保留1点可供活动
- 在家的一般活动（看电视、玩游戏）不消耗行动点，但影响成绩

#### 2.11.5 走读与住校

**当前设计**
- 走读/住校在玩法上无本质区别
- 回家后的限制主要在行动点系统
- 周内不能出学校玩，但可以选择回家

**回家功能**
- 电脑读档入口（核心功能）
- 未来扩展：玩游戏、去商场购物（需朋友邀请）

#### 2.11.6 周循环与特殊事件

**周循环**
- 自然周为单位
- 每周可能有特殊事件
- 造梦主梦境定期触发（周末）

**月考/学期节点**
- 月考影响清北班淘汰
- 成绩系统与学习次数关联
- 具体时间节点根据剧情需要设定

### 2.12 成绩系统详解

#### 2.12.1 系统定位

| 项目 | 说明 |
|------|------|
| **核心定位** | 剧情分支条件，轻量级系统 |
| **呈现形式** | 名次制（年级第X名） |
| **考试触发** | 剧情驱动，无考试场景 |
| **UI入口** | UI面板特定位置 + 学习选项悬停显示 |

#### 2.12.2 成绩数值系统

**内部计算**
- 成绩以"学习积分"形式后台计算（0-100）
- 转换为名次：`名次 = 基础排名 + (100 - 学习积分) * 随机因子`
- 加入随机因子模拟考试波动

**自然回落机制**

| 时间 | 回落速度 | 说明 |
|------|----------|------|
| 每天不学习 | -2 学习积分 | 自然遗忘 |
| 连续学习 | 无回落 | 保持状态 |

#### 2.12.3 学习行为系统

**学习方式**

| 学习方式 | 行动点消耗 | 效果 | 说明 |
|----------|-----------|------|------|
| 晚自习 | 1点 | +5 学习积分 | 留校学习 |
| 回家学习 | 不消耗 | +3 学习积分（概率） | 有概率熬夜影响第二天 |
| 和朋友一起学习 | 1点 | +6 学习积分 | 社交+学习，效率更高 |

**熬夜惩罚**
- 在家玩电脑/看电视 → 50%概率熬夜 → -3 学习积分

**设计理念**
- 核心是行动点分配：把行动点给学习就能保持成绩
- 成绩自然回落，需要定时学习
- 玩乐行为不算直接扣分，只是没有用于学习
- 熬夜是直接惩罚

#### 2.12.4 考试与排名

**考试触发**
- 月考、期中、期末：剧情节点触发
- 无考试场景，仅显示结果

**考试结果呈现**
- 弹窗显示名次
- 显示是否保留清北班资格

#### 2.12.5 清北班机制

**进入条件**
- 第一章结束自动进入清北班（剧情安排）

**保留条件**
- 期末加权成绩排名前30

**淘汰流程**

```
期末考试 → 排名低于30
    ↓
第一次：警告 + 提示"下次将淘汰"
    ↓
第二次：Game Over → 返回读档点
```

**加权制**
- 学期内所有统一考试加权平均
- 期末考试权重最高
- 决定是否淘汰

#### 2.12.6 积分奖励

**奖励规则**

| 名次 | 奖励积分 | 说明 |
|------|----------|------|
| 前5名 | 50积分 | 最高奖励 |
| 6-10名 | 30积分 | 较高奖励 |
| 11-20名 | 15积分 | 中等奖励 |
| 21-30名 | 5积分 | 达标奖励 |
| 30名后 | 0积分 | 无奖励 |

**设计理念**
- 前5和前10奖励多，之后稀释
- 学校行为，只看该次考试（非加权）

#### 2.12.7 补课支线

**支线背景**
- 原型：数学不太好，母亲叫去补课，和同地区同学一起去上课

**触发条件**
- 时间点：第二或第三章某节点
- 触发方式：母亲打电话询问
- 玩家选择：去/不去

**支线内容**
- 和同地区但不认识的同学一起补课
- 补课期间发生故事
- 解锁后角色加入世界

**角色互动**
- 触发世界任务
- 偶尔偶遇

#### 2.12.8 UI设计

**成绩查看方式**

| 方式 | 说明 |
|------|------|
| **UI面板特定位置** | 独立的成绩标签页 |
| **学习选项悬停** | 选中学习选项时显示当前名次 |
| **考试结果** | 考试后弹窗显示名次 |

**不常驻原则**
- 平时不显示成绩，保持焦点集中
- 仅在需要时显示

**成绩面板内容**

#### 2.12.9 技术实现

**GradeManager.gd (autoload)**

```gdscript
extends Node

var study_points: int = 50  # 学习积分 (0-100)
var current_rank: int = 30  # 当前名次
var exam_history: Array = []  # 考试历史记录
var qingbei_warning: int = 0  # 警告次数

signal rank_changed(new_rank)
signal qingbei_status_changed(status)  # "warning" / "eliminated" / "safe"

func study(amount: int):
    study_points = clamp(study_points + amount, 0, 100)
    update_rank()

func daily_decay():
    study_points = max(0, study_points - 2)

func update_rank():
    var random_factor = randf_range(0.8, 1.2)
    current_rank = int(15 + (100 - study_points) * random_factor)
    rank_changed.emit(current_rank)

func trigger_exam() -> Dictionary:
    var result = {
        "rank": current_rank,
        "reward": calculate_reward(current_rank),
        "qingbei_status": check_qingbei(current_rank)
    }
    exam_history.append(result)
    return result

func calculate_reward(rank: int) -> int:
    if rank <= 5: return 50
    elif rank <= 10: return 30
    elif rank <= 20: return 15
    elif rank <= 30: return 5
    return 0

func check_qingbei(rank: int) -> String:
    if rank <= 30:
        qingbei_warning = 0
        return "safe"
    else:
        qingbei_warning += 1
        if qingbei_warning >= 2:
            return "eliminated"
        return "warning"
```
